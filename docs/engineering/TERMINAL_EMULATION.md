# Терминальная эмуляция

Этот документ фиксирует первое архитектурное решение перед внедрением рабочего интерактивного shell в `mtrm`.

## Выбранная библиотека

Для первой реализации терминальной эмуляции в `mtrm` выбирается `vt100`.

Причины выбора:

- она уже решает именно ту задачу, которой сейчас не хватает проекту: принимает поток байтов терминала и поддерживает экранное состояние;
- она заметно проще для интеграции, чем полноценные терминальные движки уровня отдельного эмулятора терминала;
- она дает более высокий уровень, чем голый ANSI-парсер, поэтому не заставляет строить весь экранный слой с нуля;
- для первой рабочей версии `mtrm` важнее быстро получить корректный prompt, ввод и базовое экранное поведение, чем сразу брать максимально тяжелую основу.

## Граница ответственности

Новый поток данных должен быть таким:

1. `mtrm-process` управляет PTY и отдает сырые байты вывода.
2. Новый слой терминального экрана принимает эти байты и обновляет экранное состояние конкретной панели.
3. `mtrm-tabs` хранит живые процессы панелей вместе с их экранным состоянием.
4. `mtrm-ui` рисует уже не сырой текст, а рассчитанное состояние видимой части терминального экрана.
5. `app` координирует цикл событий, но не владеет моделью вывода панели как строкой.

## Что не меняется

- `mtrm-process` не должен превращаться в терминальный эмулятор;
- `mtrm-ui` не должен разбирать ANSI-последовательности сам;
- публичный метод `ShellProcess::current_dir()` не меняется;
- PTY остается единственным источником байтового вывода shell.

## Новая архитектурная граница

Между `process` и `ui` добавляется отдельный слой экранного состояния панели.

Этот слой:

- знает размер терминала панели;
- принимает байты из PTY;
- обновляет курсор, содержимое ячеек и видимую часть экрана;
- отдает подготовленное представление для отрисовки.

Этот слой не должен:

- запускать процессы;
- читать клавиатуру;
- знать про вкладки;
- напрямую работать с `ratatui`.

## Почему не `vte`

`vte` слишком низкоуровнев для текущей задачи.

С ним пришлось бы отдельно проектировать и реализовывать собственное экранное состояние, а это именно тот слой, который сейчас нужно получить как можно быстрее и с меньшим количеством новых ошибок.

## Почему не более тяжелый движок

Библиотеки уровня полноценного терминального эмулятора дают больше возможностей, но сейчас это преждевременно.

Для `mtrm` на этом этапе нужен не максимально общий терминальный движок, а практичный экранный слой, который:

- корректно показывает prompt;
- позволяет видеть ввод;
- переживает resize;
- не взрывает текущую архитектуру workspace.

## Текущее состояние

Текущая реализация этого слоя находится в:

- [crates/terminal_screen](../../crates/terminal_screen)
- [crates/terminal_screen/README.md](../../crates/terminal_screen/README.md)

Что уже сделано:

- у каждой живой панели в `tabs` есть собственный `TerminalScreen`;
- `app` больше не хранит `pane_text`;
- `ui` больше не рисует одну строку `text: String`;
- `PaneView` теперь содержит экранные линии и позицию курсора;
- курсор активной панели отображается визуальным выделением текущей ячейки.
- `TerminalScreen` теперь отдает не только `text`, но и terminal-cell метаданные, включая wide/continuation semantics.
- `ui` теперь рисует pane content напрямую как terminal-cell grid, а не через обычный текстовый paragraph.
- shell по умолчанию теперь запускается в интерактивном режиме, чтобы стартовый вывод и вводимая строка были видны сразу.
- для `ui` экранные линии и курсор панели отдаются через `tabs`.

## Ограничения текущей версии

- для копирования и некоторых вспомогательных проверок все еще используется текстовое представление панели, собранное поверх экранного состояния;
- курсор пока показывается простым визуальным выделением ячейки, а не отдельным режимом курсора терминала;
- качество интерактивности уже заметно лучше, но это еще не полноценный отдельный эмулятор терминала уровня специализированных программ.

## Что еще может понадобиться позже

- убрать оставшуюся промежуточную текстовую адаптацию там, где она больше не нужна;
- при необходимости усилить модель курсора и атрибутов экрана;
- отдельно решить, нужен ли проекту более глубокий уровень эмуляции beyond текущей достаточной реализации.
