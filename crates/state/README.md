# `mtrm-state`

`mtrm-state` читает и записывает снимок состояния `mtrm` на диск.

## Публичные функции

- `load_state() -> Result<Option<SessionSnapshot>, StateError>`
  - вычисляет стандартный путь через `mtrm-config`;
  - сначала пытается прочитать `state.yaml`;
  - если `state.yaml` отсутствует, использует legacy fallback на `state.toml`;
  - возвращает `Ok(None)`, если оба файла отсутствуют.
- `save_state(snapshot: &SessionSnapshot) -> Result<(), StateError>`
  - обеспечивает существование `~/.mtrm`;
  - записывает снимок в стандартный файл состояния.
- `load_state_from_path(path: &Path) -> Result<Option<SessionSnapshot>, StateError>`
  - читает снимок из произвольного пути;
  - возвращает `Ok(None)`, если файла нет.
- `save_state_to_path(path: &Path, snapshot: &SessionSnapshot) -> Result<(), StateError>`
  - записывает снимок в произвольный путь.

## Формат файла состояния

Основной формат хранения зафиксирован как `YAML`.

При записи используется YAML-файл с версией формата `0.0.1`.

На диске сохраняются:

- `version: 0.0.1`
- поля самого `SessionSnapshot`

Legacy `TOML` поддерживается только для чтения старых файлов состояния.

## Атомарная запись

`save_state_to_path()` пишет состояние атомарно в два шага:

1. Сериализует снимок и записывает его во временный файл рядом с целевым.
2. Заменяет целевой файл через `rename`.

Временный файл создается рядом с `state.yaml`, но не имеет фиксированного имени.

Он получает уникальный суффикс на каждый вызов сохранения, чтобы уменьшить риск конфликтов и сделать схему безопаснее, чем один и тот же предсказуемый путь.

Такая схема уменьшает риск получить частично записанный итоговый файл, если запись будет прервана между началом и завершением операции.

## Долговечность записи

Помимо логической атомарности, реализация теперь делает best-effort синхронизацию записи на диск:

1. после записи временного файла вызывается синхронизация самого файла;
2. после `rename` вызывается синхронизация каталога, в котором лежит `state.yaml`.

Это не делает хранение магически неуязвимым ко всем сбоям любой файловой системы, но усиливает гарантию того, что уже записанное состояние действительно переживет внезапное завершение процесса или сбой системы.
